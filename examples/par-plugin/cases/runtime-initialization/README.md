# Runtime Initialization Attribution

This direct Node.js control separates generic `Worker` creation from native-binding import and the complete Rolldown package import. It is an attribution harness, not a build benchmark.

The formal matrix freezes five controls—bare Worker, retained parent binding with empty workers, per-worker binding import, retained parent package with empty workers, and per-worker complete package import—at exactly one, two, four, and eight workers for ten rotated blocks. It uses a five-millisecond in-process sampler and forbids the perturbing operating-system thread sampler. Binding or package preload matches the normal direct-Rolldown shape: the process-level napi runtime already exists before parallel-plugin workers import the same addon. Binding-only and package-import deltas separate worker environment/module loading from generic Worker creation.

The original source-derived hypothesis was that every Node worker environment reran Rolldown's Rust `module_init`, constructed another 18-thread Tokio runtime, and discarded it because napi retains one process-global runtime. Instrumented worker-one and worker-four direct builds falsified the per-worker-constructor hypothesis: each process emitted one constructor record and no stopped-thread callback. Inspection of `napi-derive` 3.5.9 confirms that `#[napi_derive::module_init]` expands through `napi::ctor::declarative::ctor!`, a dynamic-library constructor rather than a per-environment registration callback. The callback counts recorded immediately after runtime construction and custom-runtime registration are scheduling snapshots: local observations range from 6/7 to 17/18 started threads even though the configured capacity is 18, so they cannot establish a final retained thread count. Worker import cost must therefore be attributed elsewhere; the harness requires zero or one constructor record per process, monotonic started counts bounded by 18, and zero stopped counts. A later quiesced native snapshot is required for a retained-thread-count claim.

The worker records entry and dynamic-import boundaries with the process-wide Node monotonic clock. The parent records constructor, online, ready, termination, process/main/worker CPU, V8 heaps, and whole-process RSS. Every timestamp is checked against the recorded parent or worker `performance.timeOrigin`, and every preload, worker-local snapshot, and termination boundary is checked against its surrounding process snapshots. An immediate process snapshot defines all-ready before the asynchronous worker CPU/heap queries; those later worker values are reported separately and are not subtracted from the earlier boundary. Workers take an explicit post-GC snapshot before termination, then the parent records post-termination and post-GC state. Whole-process RSS differences are controlled observations, not worker ownership. Total process and main-thread CPU are derived exactly from the first and final cumulative snapshots; residual CPU includes worker termination plus all unmeasured runtime and native work, must remain nonnegative, and is never labelled Rolldown Rust CPU.

An optional macOS `/bin/ps -M` thread sampler exists only for diagnostic smoke. It synchronously forks and cannot provide a one-millisecond cadence; enabling it perturbs short initialization and its elapsed values are invalid. Formal attribution disables it. The native module-init callback counters establish one process-level constructor invocation and immediate bounded start/no-stop behavior without process polling; they do not supply a final retained thread count.

The matrix runner pins the complete attribution distribution, runtime checkout, Node executable bytes, pool environment, harness sources including the exact worker entry, package manifest, lockfile, installed pnpm layout, and the full resolved `@rolldown/pluginutils` package used by the Rolldown entry. Formal runs require both the harness checkout and the separate attribution-runtime checkout to be clean and stable before and after the matrix, reject inherited `NODE_OPTIONS` and every Node module-compile-cache control, set `NODE_ENV=production`, admit the host before every child, reject any pageout or swapout, and recheck the immediate host gates after every child. Each two-block pair uses one rotation offset and reverses the odd block, so every named contrast executes five times in each order. Worker readiness, resource replies, termination, and the complete child have fixed timeouts; failure cleanup terminates or unreferences every created worker. Prepare the exact attribution checkout with the frozen lockfile, then run the canonical untimed smoke or formal matrix explicitly:

```sh
(cd /path/to/attribution-runtime && vp install --frozen-lockfile --prefer-offline --ignore-scripts)
ROLLDOWN_RESEARCH_PACKAGE_ROOT=/path/to/attribution-runtime/packages/rolldown /path/to/node-v24.18.0/bin/node ./run-matrix.mjs ./smoke-matrix.json ./.results/smoke.raw.json
ROLLDOWN_RESEARCH_PACKAGE_ROOT=/path/to/attribution-runtime/packages/rolldown /path/to/node-v24.18.0/bin/node ./run-matrix.mjs ./formal-matrix.json ./.results/formal.raw.json
/path/to/node-v24.18.0/bin/node ./summarize-matrix.mjs ./.results/formal.raw.json ./.results/formal.summary.json
```

`.results/` is committed to the harness ignore rules, so writing the mandatory formal raw there does not invalidate the clean-worktree summary gate. `summarize-matrix.mjs` rejects an output that aliases its raw input, rechecks that the raw bytes did not change, and atomically replaces the summary only after validation. It independently revalidates the exact matrix, order-balanced rotation, child, host, runtime, package environment, current Node artifact, external peak RSS, and worker source; it binds the summary to the raw byte length and SHA-256. It reports parent preload, constructor dispatch, first/all online, first/all worker entry, first/all ready, ready skew, per-worker and critical-path import, termination, process/main/worker/residual CPU, ready and retained RSS, isolate heap, and module-initialization thread and duration distributions. Binding and package-import deltas use within-block pairs. The additional JavaScript package layer uses the within-block difference-in-differences `(package import - retained package) - (binding import - retained runtime)`, so parent preload state is cancelled. Every paired metric includes a deterministic 100,000-resample median bootstrap interval with seed `0x20260712`.

The generic controls deliberately do not measure a real plugin factory/configuration lifecycle, stage-local GC events, page faults, or I/O. Attribution runtime `41833e129` separately preserves the existing metrics-off initializer and static worker entry while giving metrics mode its own launcher; it records the combined runtime/native-binding import, worker implementation import, factory, bindingification and registration, matching ordinary JavaScript bindingification, the complete outer native normalization boundary, correlated lifecycle records, and explicit asynchronous CPU bounds. Existing Vue and MDX traces still need to consume that runtime on the quiet host. Their per-worker timelines can derive observed service for calls 1, 2, 4, 8, 16, and 32 and calls after 32, but that cold-to-steady difference is not by itself proof of JIT because the real sources have different compile costs.

## Durable evidence promotion

`.results/` is staging state only and is never conclusion evidence. After a successful formal run, copy the exact raw and summary bytes into the research repository's canonical content-addressed initialization-evidence directory and create a compact pointer containing their relative paths, byte lengths, SHA-256 hashes, the harness commit and source-manifest hash, the attribution-runtime commit/distribution/binding/package-entry hashes, and the Node binary hash. The raw, summary, and pointer must all be tracked and byte-identical to `HEAD`; the pointer must not escape the repository; and a fresh-clone verifier must recompute the pointer from the raw report and summary before any result document consumes it. The controlled Vue `evidence-verifier.mjs` implementation is the reference for tracked `git show HEAD:<path>` equality and canonical raw-path checks. Do not mark a staged `.results/` file as promoted, and do not create a pointer until formal quiet-host evidence exists.

Formal repeated initialization attribution requires the same restarted quiet-host gate and artifact provenance as the scale protocol. Its elapsed fields must not be mixed into uninstrumented wall curves.
