# Parallel Plugin Initialization Attribution Contract

This research-only contract closes the initialization stages that were previously hidden by the static parallel-worker entry. It does not turn instrumented elapsed values into wall benchmark evidence.

This follow-on schema starts from Rolldown attribution revision `41833e129`. Its metrics-off byte pins intentionally still name the earlier `8e35a2249` attribution base because that is the revision where those untouched functions and the static worker entry were frozen.

The metrics-off worker still uses the original static `parallel-plugin-worker` entry and module graph. Its main-thread `initializeWorkers` and `initializeWorker` functions remain byte-identical to attribution base `8e35a2249`; in particular, the research work does not add or remove `unref`, listener, timeout, error, exit, or readiness semantics on that path. The contract pins hashes for both initializer functions and the static worker entry. Metrics mode alone selects `parallel-plugin-worker-metrics`, a separate research launcher whose first Rolldown-owned statement records `launcherEntryAt`. It then imports the metrics runtime, captures the worker-isolate baseline, and dynamically imports the compiled worker-runtime graph. That graph statically imports `binding.cjs`, so `runtimeAndBindingImport` includes JavaScript graph loading/evaluation and native-addon loading. Those parts cannot be separated honestly without changing the production module boundary or adding loader-level Node instrumentation. The GC observer begins after `node:perf_hooks` becomes available, so earlier GC cannot be recovered.

`createBundlerOptions` allocates one research-only `metricsId` and propagates it through pool initialization, every worker launcher/bootstrap record, both pool lifecycle records, and the native binding record. Its record covers metrics-runtime setup, plugin-option normalization, the `outputOptions` hook, final ordinary/parallel/builtin plugin normalization, parallel-pool initialization, plugin-context construction, per-plugin `bindingifyPlugin` work inside `bindingifyInputOptions`, the remaining input bindingification, and output bindingification. Its first main-isolate resource snapshot necessarily follows metrics-runtime setup because the heap/GC observer does not exist earlier. Later process snapshots report whole-process RSS and CPU plus main-isolate heap and GC. Per-plugin elapsed values do not claim to isolate native N-API materialization.

Both native call paths now use the same outer `BindingBundlerOptions` normalization helper. The native total begins after options destructuring but before parallel-registry transfer, includes `WorkerManager` construction, binding-option normalization, plugin conversion, and `BundlerConfig` construction, and ends synchronously before `ClassicBundler::create_bundle` and scan. The record exposes registry transfer, manager construction, binding-option normalization, and nested plugin-materialization durations instead of calling the nested plugin duration the whole native boundary. It deliberately excludes JavaScript bindingification, scan, hooks, and the rest of the build. It is not a general production API.

Worker bootstrap now records synchronous resource boundaries before implementation import, between implementation import and factory, between factory and `bindingifyPlugin`, and between bindingification and registry registration. A final pair brackets `registerPlugins`. Each boundary contains whole-process CPU/resource usage/RSS, current-worker thread CPU, current-worker V8 heap/event-loop utilization, and GC entries observed by that worker. Windows refer to the named boundaries rather than copying snapshots, contain their wall stage and exact counter deltas, and state that the snapshots include boundary-capture gaps. Process CPU and RSS also include concurrent work elsewhere in the process; only worker-thread CPU and worker-isolate heap/GC have worker-local scope. Signed RSS and heap changes are observations, never plugin, factory, worker, isolate, or stage ownership.

Lifecycle process snapshots report synchronous `captureStartedAt` and `captureFinishedAt` bounds because the individual CPU and RSS read instants are not exposed. Initialization preserves the all-workers-ready and post-ready-resource boundaries. Termination preserves the pre-termination worker snapshots and the whole-process post-termination boundary. Process CPU snapshots and asynchronous `Worker.cpuUsage()` reads do not have identical endpoints. The lifecycle record therefore exposes an outer process window, an inner process window when available, and explicit earliest/latest bounds for every worker sample. It never subtracts those skewed windows into a claimed Rust/native CPU residual or labels them exact attribution.

After workers terminate and the native bundler closes, metrics mode requests two parent-isolate GCs and emits one separate post-close record. The record reuses the termination report's `afterTermination` snapshot and joins it to `afterBundlerCloseBeforeParentGc` and `parentPostGc`, with synchronous capture bounds, whole-process/main-thread CPU deltas, signed RSS observations, and the requested/available/executed GC counts. The main contract matrix starts Node with `--expose-gc` and requires both passes; an additional child starts without it and requires unavailable/zero-pass semantics. Other callers likewise record GC as unavailable instead of claiming that collection occurred. `parentPostGc` is an observation after explicit requests, not proof that V8 or the native allocator released every reclaimable page.

Metrics-mode reporting waits for each stderr record to flush before the next native boundary can write its own record. This matters because the bootstrap report can exceed an operating system pipe's atomic-write size; an asynchronous write allowed the later native registration record to interleave with and corrupt the JSON line.

The validators require one `epochMs - monotonicMs` clock origin for every timestamp within an isolate, per-plugin import → factory → bindingification order within bootstrap, named resource-window references and arithmetic that exactly match their shared boundaries, registration containment, worker-local snapshots contained by bootstrap, worker launcher/bootstrap/pool/plugin-index agreement, main-thread constructor → `online` → ready-message order, lifecycle process and asynchronous resource snapshots in their declared order, CPU windows equal to their lifecycle snapshot bounds, and worker timestamps contained by the main constructor/ready window. The post-close validator also requires two requested GC passes, availability-consistent execution, termination → native close → parent GC ordering, exact CPU arithmetic, exact RSS fields, and endpoint/ownership limitations. The negative contract shifts epoch values, changes deltas and ownership text, changes boundary references, and reverses or moves snapshots outside their boundaries to prove these checks fail. A validation or metrics-reporting failure after worker creation terminates every created worker before propagating the error.

After a release binding and distribution have been built with Node.js 24.18.0, run the synthetic negative checks and real ordinary/parallel metrics-off/metrics-on smoke with:

```sh
/path/to/node-v24.18.0/bin/node --no-warnings=ExperimentalWarning examples/par-plugin/verify-init-metrics-contract.mjs
```

The real smoke requires exact record counts, `metricsId` correlation across JavaScript, pool, worker, Rust, termination, and post-close records, exact JavaScript/Rust plugin count and index correlation, two complete worker launcher/bootstrap/lifecycle records, one post-close record, and identical output and lifecycle counters with metrics disabled and enabled. It also injects failures immediately after worker creation and immediately after pool initialization; each child catches the error and must exit normally, proving that both cleanup paths terminate their workers. It freezes the current lifecycle behavior as well: the ordinary plugin receives one `closeBundle`, while the parallel plugin receives none. It is a correctness and contract test, not a performance run.

This round covers one `generate` or `write` build followed by `close`, plus the existing `scan` cleanup path. Repeated `generate`/`write`, watch, rebuild, HMR, and dev-engine lifecycles are deliberately not part of this contract.

Irreducible boundaries remain explicit: Node does not expose exact worker or synchronous process counter read instants; RSS and process CPU are shared; V8 heap statistics exclude native/shared memory; native-addon loading cannot be split from worker runtime graph evaluation at the current module boundary; the first GC observer cannot recover events that preceded its import; explicit GC does not guarantee allocator page release; and instrumented stage elapsed/resource values remain diagnostic rather than uninstrumented wall evidence or ownership attribution.
